## Авто-заполнение Table Results после закрытия смены

### Проблема
За 12 мая в `table_daily_results` нет строк. Функция `populate_table_daily_results_for_day` существует, но никем не вызывается. Кнопки делать не будем — всё автоматически.

### Решение (1 миграция, без UI)

**Шаг 1. Backfill за 12 мая**
```sql
SELECT populate_table_daily_results_for_day(
  '48f4404f-7724-418c-8365-29af3998e113'::uuid, '2026-05-12'::date);
```
(плюс пройти по последним 30 дням на случай других пропусков — безопасно, UPSERT)

**Шаг 2. Авто-вызов после закрытия cage-смены**
Расширить существующую функцию `apply_cage_shift_closing()` (или добавить отдельный AFTER UPDATE триггер на `shifts` при переходе `status → 'closed'`):
```sql
PERFORM populate_table_daily_results_for_day(
  NEW.casino_id,
  business_date_of(NEW.opened_at)
);
```
Так результаты появляются сразу после закрытия каждой смены — не ждём закрытия бизнес-дня. UPSERT перезапишет, если потом будет ещё одна смена за тот же день.

**Шаг 3. Дублирующая страховка на закрытие бизнес-дня**
Триггер `AFTER INSERT ON business_day_closures` → `populate_table_daily_results_for_day(NEW.casino_id, NEW.business_date)`. Идемпотентно, перекроет крайние случаи (ручное закрытие без cage-смены, авто-закрытие в 11:00).

**Шаг 4. Версия**
`package.json` → 1.0.152.

### Почему так
- Чистый backend-фикс, без UI/кнопок.
- Двойная страховка (после смены + после закрытия дня) — не пропустит даже если кто-то закроет день вручную без последней смены.
- UPSERT по `(casino_id, table_id, business_date)` безопасен при повторных вызовах.
